# Section 1: What This Domain's Data Actually Means
Usage & Performance data is the ultimate arbiter of engineering success. The Analytics agent must aggressively separate "User Preferences" (e.g., "I wish it came in blue") from "Structural Failures" (e.g., "I cannot figure out how to turn it on"). If a high percentage of users fail the Primary Task, the product is fundamentally unviable for the mass market, regardless of how much they like the aesthetic. The Analytics agent must synthesize the transcripts to identify the exact moments in the UI/Hardware where human intuition fails.

# Section 2: Coverage Interpretation Guide
**The Task Completion Matrix:**
- Pass without friction: The ideal state. Highly intuitive.
- Pass with friction/manual: Commercially viable, but will suffer from high customer support costs.
- Complete Failure: A fundamental product emergency. Requires a hardware or software redesign.

# Section 3: Quality Weighting Rules
**The 'Workaround' Multiplier:** If a user invents a "hack" or a workaround to force the product to do what they want, this data is weighted at 3x. User workarounds point directly to missing features that the engineers failed to anticipate.

# Section 4: Benchmark Context
**The Manual Dependency:** The Analytics agent must calculate the "Manual Dependency Rate." If 80% of users who successfully completed the task only did so *after* reading the instruction manual, the product's physical design is highly unintuitive and will likely generate negative early reviews.

# Section 5: Output Format Specification
**Section 1: Study Parameters**
Task scope, environmental protocols, cohort expertise.
**Section 2: Executive Summary**
The definitive ruling on the product's functional viability (The Pass/Fail rate).
**Section 3: Primary Task Diagnostic**
Detailed breakdown of exactly where the usage sequence breaks down.
**Section 4: The 'Friction & Hack' Report**
Synthesis of the unintended difficulties and the user-invented workarounds.
**Section 5: Decision Map Response**
Direct mapping to the UI redesigns, hardware modifications, or instructional updates requested in the brief.
**Section 6: Limitations and Confidence Notes**
Required section.

# Section 6: Multi-Session Analysis Guide
Analyze for "Sequential Cascades." Does failing Step 2 automatically guarantee that the user will break the machine in Step 3? Identify these cascading failure points, as they represent the highest liability for product returns.

# Section 7: Flagging and Limitation Language
**When the primary task fails systematically:**
"Synthesis of the usage sequences reveals a catastrophic failure in the product's core utility. [Percentage]% of respondents were unable to complete [Primary Task] without direct intervention or consultation of the manual. The primary failure point was consistently isolated to the [Specific Mechanism/Feature]. This is not an aesthetic preference; it is a structural barrier to operation. Advancing to mass production with this interface will result in an astronomically high volume of customer support tickets and retail returns. We recommend an immediate redesign of the [Mechanism]."
