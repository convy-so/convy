# Section 1: Domain Identity
Software Experience Research goes beyond the mechanical UI (Usability) to evaluate the core utility of a digital product. It answers the question: *Is this software actually making the user's life or job better?* It focuses on workflow integration, perceived value, feature relevance, and the psychological relief (or burden) generated by adopting the software. It is the primary domain for finding product-market fit in SaaS, consumer apps, and enterprise platforms.

# Section 2: Research Objectives This Domain Can and Cannot Answer
**Can answer:**
- Whether the software solves a high-priority problem for the user
- How the software integrates into the user's existing daily workflow or "tech stack"
- Which features are considered "Mission Critical" vs "Nice to Have"
- The primary catalyst that would cause a user to cancel their subscription (Churn Risk)

**Cannot answer:**
- Exactly which button they clicked when trying to reset a password (requires Usability & UX)
- The technical difficulty of integrating the API with a third-party CRM (requires Platform Ecosystem)

# Section 3: Brief Interrogation Guide
**The Value Proposition:**
- What is the *core promise* the software makes? (e.g., "Saves you 5 hours a week on accounting"). The brief must anchor the research around proving or disproving this specific promise.

**The Workflow Reality:**
- What did the user use *before* this software? (e.g., Excel, pen and paper, a competitor). The AI must mandate that the client defines the baseline alternative.

# Section 4: Decision Map Interrogation
**Questions to ask:**
- If users state that the software does exactly what it promises, but they still don't use it because it requires too much manual data entry, will you invest in automation features?
- If the "flagship" AI feature is completely ignored by 80% of users, will marketing stop leading with it in ads?

**Well-formed decision map example:**
> Experience outcome: If the 'Time-Saving' utility is verified but users complain about data-entry friction, the product roadmap will prioritize automatic bank-feed integrations. If the core promise is invalidated (users don't actually care about saving time on this task), product strategy will pivot to highlighting 'Compliance & Accuracy' instead.

# Section 6: Audience Model Interrogation Guide
**Psychographic dimensions:**
- Mandated vs Voluntary User: The brief must establish if the user *chose* to buy this software (Consumer/Freelancer) or if it was *forced* upon them by their boss (Enterprise Employee). The psychology of a mandated user is inherently more hostile and requires different probing tactics.

# Section 7: Constitutional Constraints
1. **The 'Aesthetics' Filter.** The AI must explicitly instruct the Conducting agent to ignore feedback about colors, fonts, or minor branding choices. If the respondent complains about the font, the AI must redirect them to whether the *information* provided by the font was actually useful.

# Section 10 — Cross-Domain Signals
## Outbound signals
- **Micro-friction explosion:** If the user loves what the software does, but spends the entire interview raging about how hard it is to navigate the menu, this flags **Digital Product: Usability & UX**.

# Section 11 — Supplementary Brief Questions
**The single required supplementary question:**
"What is the most common reason users currently give when they cancel or uninstall this software?"

**Initialization mandate:**
The answer populates parameters required by the `BRIDGE-dpse-*` node to anticipate the primary Churn Risk vector.
